SmartPolicy · an authorization protocol

The rules live on-chain. The code never has to.

Define what an agent, a user, or a contract is allowed to do — once, as a policy in an immutable registry. Enforce it in smart contracts, in backends, and over MCP. Change the rules without redeploying anything.

§ 1

The problem

Every team shipping AI agents hits the same wall: the agent can technically do anything its keys allow, and the only ways to constrain it are hardcoded prompts or redeployed code. Smart-contract teams hit the mirror image — access rules baked in at deploy time.

SmartPolicy separates the rule from the enforcement point. A policy is data, not code: members, admins, per-action rules, conditions, expiry. Changing a rule is a transaction, not a release.

§ 2

One policy, three enforcement points

Everything that consumes the protocol asks one canonical question: “may subject S perform action A under policy N?”

a

Smart contracts

Inherit PolicyGate, add one modifier. Membership and action rules are checked with a free registry view call.

b

AI agents, over MCP

Agents call the MCP server — policy_check, grant_issue — and receive a short-lived EIP-712 signed grant as portable proof.

c

Backends & APIs

Verify a grant with one signature check instead of building a permission system. Grants are single-use, chain-bound, and target-bound.

§ 3

What a policy can express

The on-chain registry enforces three things directly, with no code to trust but its own:

i

Who

Member and admin sets. Open membership if you want everyone; a curated list if you don’t. Adding or removing an address is one transaction.

ii

What

Per-action rules on free-form action names: MEMBERS, ANYONE, NOBODY. Granularity is yours — withdraw.small and withdraw.large can carry different rules.

iii

When

Policy expiry, and an irreversible lock that freezes the rules while keeping them enforceable.

Everything richer — spending limits, budgets, business hours, external facts — lives in grant issuance: the policy owner authorizes an issuer (ours, or one they run), and that issuer applies any logic it likes before signing a short-lived, single-use, target-bound grant. Flexible rules off-chain; incorruptible enforcement on-chain. The rule brain can be as smart as your code — the bouncer never bends.

§ 6

Integration is one inheritance

Solidity — protect any function

contract Treasury is PolicyGate {
    constructor(address registry, address verifier)
        PolicyGate(registry, verifier) {}

    function pay(address to, uint256 amount)
        external
        onlyAllowed(POLICY_ID, "treasury.pay")
    { /* … */ }
}

MCP — zero-config, any agent

npx smartpolicy-mcp        # reads run against the public testnet registry

§ 7

Trust properties

The registry asks to sit in your authorization path. That demands more than a pitch — these hold by construction:

  1. Immutable core.
    No proxy, no upgrade path, no admin who can change the rules of the rule system. New protocol versions are new deployments.
  2. No token.
    Revenue is small flat fees on policy writes and metered (x402) calls on the hosted API. Reads and verification are free, forever.
  3. Your issuers, not ours.
    Policy owners choose which addresses may sign grants. The hosted service is one option, never a privileged one.
  4. Verify, don’t trust.
    Contracts are source-verified on-chain; the protocol owner’s only powers are fee parameters, hard-capped at deploy.
